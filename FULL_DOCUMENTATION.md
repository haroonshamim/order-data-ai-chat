# Order Data AI Chat - Full Documentation

This document consolidates and organizes the comments from:
- `client/src/components/ChatInterface.jsx`
- `server/server.js`

It explains the architecture, flow, and purpose of each major part in a clear way.

## 1. Project Overview

The app is a chat-based analytics assistant for order data.

- The frontend lets users ask natural language questions.
- The backend converts questions to SQL using Gemini.
- Order data is fetched from Supabase.
- A local executor interprets common SQL patterns and returns results.
- Gemini then generates a natural-language answer from those results.

## 2. Frontend Documentation (`ChatInterface.jsx`)

## 2.1 Imports and Their Purpose

- `React, useState, useRef, useEffect`
: Used to build the component, manage state, track DOM references, and run side effects.

- `axios`
: Sends HTTP requests from frontend to backend (`POST /api/chat`).

- `./ChatInterface.css`
: Applies styling for chat layout and message design.

## 2.2 Component Responsibility

`ChatInterface` is the main chat UI component. It:
- Stores messages.
- Stores current input text.
- Tracks loading state.
- Sends user questions to backend.
- Displays bot responses and optional SQL query details.
- Auto-scrolls chat to latest message.

## 2.3 State Management

### `messages`
Array of message objects. Each message may include:
- `type`: `'user'` or `'bot'`
- `text`: Message content
- `sql` (optional): SQL query generated by backend
- `data` (optional): Raw result data

Initial message is a bot welcome prompt with example questions.

### `input`
String value bound to the text input field.

### `loading`
Boolean indicating whether a request is in progress.
- `true`: Disable input and button, show processing message.
- `false`: User can send a new message.

## 2.4 Ref Usage

### `messagesEndRef`
A DOM reference used as a target for smooth scrolling.

- Created with `useRef(null)`.
- Attached to an empty `<div />` at bottom of messages.
- Scrolled into view whenever messages update.

## 2.5 Helper Function

### `scrollToBottom()`
Calls:
- `messagesEndRef.current?.scrollIntoView({ behavior: 'smooth' })`

Purpose:
- Keep latest chat message visible automatically.

## 2.6 Side Effect

### `useEffect(() => { scrollToBottom(); }, [messages])`
Runs every time `messages` changes.

Purpose:
- Automatically scroll after user or bot adds a message.

## 2.7 Message Sending Flow (`sendMessage`)

1. Prevent default form submission using `e.preventDefault()`.
2. Ignore empty/whitespace-only input using `if (!input.trim()) return`.
3. Store current input as `userMessage`.
4. Append user message into `messages`.
5. Clear input (`setInput('')`).
6. Set loading state (`setLoading(true)`).
7. Send request with Axios:
   - URL: `http://localhost:5000/api/chat`
   - Body: `{ message: userMessage }`
8. Parse backend response:
   - `response` -> AI answer text
   - `sqlQuery` -> SQL generated by backend
   - `results` -> computed data
9. Append bot message with text + SQL + results.
10. On error, append bot error message.
11. In `finally`, set loading to false.

## 2.8 Rendering Structure

- Header section
  - Title: "Order Analytics AI Chat"
  - Subtitle explaining purpose

- Messages section
  - Iterates through `messages`
  - Renders `msg.text`
  - If `msg.sql` exists, shows SQL in collapsible `<details>`

- Loading indicator message
  - Visible only when `loading === true`

- Input form
  - Text input bound to `input`
  - Submit button triggers `sendMessage`
  - Input/button disabled while loading

## 2.9 Error Handling (Frontend)

If API call fails:
- Error logged to browser console (`console.error('Chat error:', error)`).
- User sees a readable bot error message in chat.

## 3. Backend Documentation (`server.js`)

## 3.1 Imports and Setup

- `express`: Web server framework.
- `cors`: Enables cross-origin requests from frontend.
- `dotenv`: Loads environment variables from `.env`.
- `createClient` from `@supabase/supabase-js`: Supabase database client.
- `GoogleGenerativeAI`: Gemini API client.

Initialization:
- `dotenv.config()` loads env variables.
- Gemini model is selected via:
  - `process.env.GEMINI_MODEL`
  - fallback: `'gemini-2.5-flash'`
- Express app uses:
  - `cors()`
  - `express.json()`
- Supabase client uses:
  - `SUPABASE_URL`
  - `SUPABASE_SERVICE_KEY`

## 3.2 Health Endpoint

### `GET /api/health`
Purpose:
- Verify backend can connect to Supabase.

Behavior:
- Queries `orders` table count.
- Returns status JSON with server/database health and order count.
- Returns `500` on connection/query errors.

## 3.3 Orders Endpoint

### `GET /api/orders`
Purpose:
- Fetch all order records.

Behavior:
- Selects all rows from `orders` table.
- Returns `{ orders: data }`.
- Returns `500` on fetch errors.

## 3.4 SQL Generation Helper

### `generateSQLFromQuestion(question)`
Purpose:
- Generate SQL from natural language using Gemini.

Prompt constraints include:
- Return only SQL query.
- Use table name `orders`.
- Date format `YYYY-MM-DD`.
- Revenue uses `SUM(total)`.
- Counts use `COUNT(*)`.

Note:
- Function exists for SQL generation logic but chat route currently performs SQL generation inline with similar prompt text.

## 3.5 Chat Endpoint

### `POST /api/chat`
Input:
- JSON body with `message` (non-empty string).

Validation:
- If invalid, returns `400`.

Main flow:
1. Receive user message.
2. Generate SQL using Gemini.
3. Fetch all orders from Supabase.
4. Execute query logic locally using `executeQueryLocally(sqlQuery, allOrders)`.
5. Ask Gemini to generate a conversational answer from the computed results.
6. Return JSON:
   - `question`
   - `sqlQuery`
   - `results`
   - `response`

Error behavior:
- SQL generation failure returns `500` with details.
- Any unhandled error returns `500` internal server error.

## 3.6 Local Query Executor

### `executeQueryLocally(sqlQuery, allData)`
Purpose:
- Interprets recognized SQL-like patterns and computes results without direct SQL execution.

Handled patterns:
- `SUM(...)` and `JANUARY`
: Filters January completed orders and returns total revenue sum.

- `COUNT(*)` and `GROUP BY`
: Groups by product and returns total quantity per product.

- `TOP 3` or `LIMIT 3`
: Aggregates spend per customer, sorts descending, returns top 3.

Fallback:
- Returns first 20 rows when query pattern does not match known rules.

Failure fallback:
- Logs error and returns first 20 rows.

## 3.7 Server Startup

- Runs on `process.env.PORT || 5000`.
- Logs URLs for:
  - server root URL
  - health check endpoint
  - orders endpoint

## 4. End-to-End Request Lifecycle

1. User types question in frontend chat box.
2. Frontend sends `POST /api/chat`.
3. Backend asks Gemini for SQL.
4. Backend fetches all `orders` from Supabase.
5. Backend computes result locally from SQL intent.
6. Backend asks Gemini to turn results into human-friendly explanation.
7. Backend returns SQL + raw result + explanation.
8. Frontend shows AI response and optional SQL details.

## 5. Environment Variables Required

Backend expects:
- `GEMINI_API_KEY`
- `GEMINI_MODEL` (optional, defaults to `gemini-2.5-flash`)
- `SUPABASE_URL`
- `SUPABASE_SERVICE_KEY`
- `PORT` (optional, defaults to `5000`)

## 6. Notes and Improvement Opportunities

- `generateSQLFromQuestion()` is currently not used by the route and can be integrated to avoid duplicated prompt logic.
- Current local query executor supports specific patterns; extending parser rules will improve coverage.
- Frontend currently displays SQL text but not tabular `results`; optionally add a result table for transparency.
